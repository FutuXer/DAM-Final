# 团队分工文档 — DAM 期末项目

> 选题 B：多源数据融合 + 5 种算法系统性对比 —— Home Credit 信用违约预测
> 团队规模：4 人

---

## 总体分工概览

| 角色 | 负责数据表 | 核心职责 |
|------|-----------|---------|
| **队员 A** | `application_train.csv` | 主表清洗 + 特征编码 + 最终宽表合并 |
| **队员 B** | `bureau.csv` + `bureau_balance.csv` | 外部征信局特征工程 |
| **队员 C** | `POS_CASH_balance.csv` + `credit_card_balance.csv` + `installments_payments.csv` + `previous_application.csv` | 内部行为时序特征工程 |
| **队员 D** | 所有表（接收宽表后） | 5 种算法训练 / 调参 / 评估 / SHAP 可解释性 + 报告统稿 |

---

## 队员 A — 主表特征工程师

### 负责数据
- `application_train.csv`（主表，~307K 行，122 列）

### 任务清单

| # | 任务 | 说明 |
|---|------|------|
| A1 | 缺失值处理 | 统计各列缺失率，对高缺失率列做删除/保留决策；数值列用中位数/均值填充，类别列用众数填充 |
| A2 | 异常值处理 | 重点处理 `DAYS_EMPLOYED` = 365243（异常值，实际含义为"无业"）；`DAYS_BIRTH` 极端值；`AMT_INCOME_TOTAL` 离群点盖帽 |
| A3 | 类别特征编码 | 对 `NAME_CONTRACT_TYPE`、`CODE_GENDER`、`NAME_EDUCATION_TYPE`、`NAME_FAMILY_STATUS`、`NAME_HOUSING_TYPE`、`OCCUPATION_TYPE` 等做 One-Hot / Target Encoding |
| A4 | 时间特征转换 | 所有 `DAYS_` 开头的列取绝对值转成年（÷365），方便解释 |
| A5 | 特征衍生 | 如收入/贷款比（`AMT_INCOME_TOTAL / AMT_CREDIT`）、年金/收入比（`AMT_ANNUITY / AMT_INCOME_TOTAL`）、年龄分箱、外部评分加权组合等 |
| A6 | 最终宽表合并 | 接收 B、C 成员产出的特征表，以 `SK_ID_CURR` 为 key 做 left join，输出最终训练矩阵 |

### 交付物
- `processed_application.csv`（清洗编码后的主表）
- `final_feature_matrix.csv`（与 B、C 合并后的完整宽表）

---

## 队员 B — 外部征信特征工程师

### 负责数据
- `bureau.csv`（~1.7M 行，17 列）
- `bureau_balance.csv`（~27M 行，3 列）

### 任务清单

| # | 任务 | 说明 |
|---|------|------|
| B1 | bureau 基础聚合 | 以 `SK_ID_CURR` groupby，统计：历史贷款总数、活跃贷款数、已关闭贷款数、各种 `CREDIT_TYPE` 的 one-hot 占比 |
| B2 | bureau 金额聚合 | 对 `AMT_CREDIT_SUM`、`AMT_CREDIT_SUM_DEBT`、`AMT_CREDIT_SUM_OVERDUE`、`AMT_CREDIT_MAX_OVERDUE` 做 mean / max / min / sum |
| B3 | bureau 时间聚合 | 对 `DAYS_CREDIT`（距今天数）做 mean / min（最近一次）/ max（最早一次），反映信用历史长度和新鲜度 |
| B4 | bureau 逾期聚合 | 对 `CREDIT_DAY_OVERDUE` 做 max / mean / sum；对 `CNT_CREDIT_PROLONG`（延期次数）做 sum / mean |
| B5 | bureau_balance 状态聚合 | 先 join bureau_balance 到 bureau，再按 `SK_ID_CURR` 聚合；统计各 STATUS（0/C/X/1/2/3/4/5）的占比、最长连续逾期月数、距当前最近 N 个月的逾期趋势 |
| B6 | 衍生特征 | 如逾期记录占比（`STATUS ≥ 1 的月数 / 总月数`）、近期逾期恶化趋势（最近 6 个月 vs 全部） |
| B7 | 输出合并 | 以 `SK_ID_CURR` 为 key，输出 bureau 特征表给队员 A |

### 交付物
- `features_bureau.csv`（以 SK_ID_CURR 为索引的外部征信特征表，预计 50-100 维）

---

## 队员 C — 内部流水特征工程师

### 负责数据
- `previous_application.csv`（~1.6M 行，37 列）
- `POS_CASH_balance.csv`（~10M 行，8 列）
- `credit_card_balance.csv`（~3.8M 行，23 列）
- `installments_payments.csv`（~13M 行，8 列）

### 任务清单

| # | 任务 | 说明 |
|---|------|------|
| C1 | previous_application 聚合 | 以 `SK_ID_CURR` groupby：历史申请总次数、批准/拒绝率、申请金额均值/总和、`DAYS_DECISION` 均值（审批速度）、各 `NAME_CONTRACT_STATUS` 占比、`CODE_REJECT_REASON` 编码 |
| C2 | POS_CASH 聚合 | 以 `SK_ID_CURR` groupby：POS 贷款笔数、活跃合同占比、`SK_DPD` 逾期天数的 max/mean、`CNT_INSTALMENT_FUTURE`（剩余期数）均值、合同状态分布 |
| C3 | credit_card 聚合 | 以 `SK_ID_CURR` groupby：平均透支率（`AMT_BALANCE / AMT_CREDIT_LIMIT_ACTUAL`）、近 6 个月平均透支额、提现/消费金额均值、还款金额/最低还款比、`SK_DPD` max/mean |
| C4 | installments 聚合 | 以 `SK_ID_CURR` groupby：实还/应还差额（欠款倾向 = `AMT_INSTALMENT - AMT_PAYMENT` 的 mean/sum）、提前还款天数均值（`DAYS_INSTALMENT - DAYS_ENTRY_PAYMENT`）、逾期次数（实还日 > 应还日） |
| C5 | 跨表衍生特征 | 如：历史申请中的批准金额总和 vs 主表当前贷款金额的比值；信用卡高频提现用户标记 |
| C6 | 输出合并 | 以 `SK_ID_CURR` 为 key，将所有内部特征合并输出给队员 A |

### 交付物
- `features_internal.csv`（以 SK_ID_CURR 为索引的内部行为特征表，预计 80-150 维）

## 队员 D — 算法与评估总监

### 前置依赖
- 接收队员 A 产出的 `final_feature_matrix.csv`

### 任务清单

| # | 任务 | 说明 |
|---|------|------|
| D1 | 数据预处理 | 再次检查缺失值（少量遗留）、标准化/归一化（供 LR 和 MLP 使用）、划分 train/val/test |
| D2 | 不平衡处理 | 配置 SMOTE 过采样 或 `scale_pos_weight` (XGB/LGB)，对比处理前后效果 |
| D3 | 特征选择 | 基于 RF 特征重要性 top-k、Pearson 相关性去冗余、L1 正则化（LR），降维至 100-200 维 |
| D4 | 模型 1：Logistic Regression | sklearn LogisticRegression，作为可解释性基线 |
| D5 | 模型 2：Random Forest | sklearn RandomForestClassifier，关注鲁棒性 |
| D6 | 模型 3：XGBoost | xgboost.XGBClassifier，调参（max_depth, learning_rate, subsample, colsample_bytree） |
| D7 | 模型 4：LightGBM | lightgbm.LGBMClassifier，调参（num_leaves, min_child_samples, feature_fraction） |
| D8 | 模型 5：CatBoost 或 MLP | CatBoost 处理类别特征优势；或 sklearn MLPClassifier 作为深度学习对比 |
| D9 | 统一评估 | 对所有模型在同一 5-Fold CV 上输出：AUC-ROC、Recall、F1-Score、Precision；绘制 ROC 曲线对比图 |
| D10 | SHAP 可解释性 | 对表现最优的树模型做 SHAP summary plot + feature importance bar chart；对 LR 做系数解释 |
| D11 | 报告撰写 | 统合 A/B/C 成员的特征工程描述，完成 10-20 页项目报告（含方法论、实验设计、结果分析、结论） |
| D12 | PPT 制作 | 制作答辩 PPT，含项目背景 → 方法 → 特征体系 → 实验结果 → SHAP 解释 → 结论 |

### 交付物
- `model_training.py` / `.ipynb`（完整训练 + 评估代码）
- `shap_analysis.py` / `.ipynb`（SHAP 可视化）
- 5 种算法对比结果表 + ROC 曲线图
- 项目报告（10-20 页）
- 答辩 PPT

## 协作规范

1. **代码仓库**：统一使用当前 GitHub/Gitee 仓库，每人建自己的分支，最终合并到 main
2. **数据路径**：所有数据文件放在 `data/` 目录下（已加入 .gitignore，不上传大文件）
3. **输出格式**：特征表统一以 `SK_ID_CURR` 为 index 的 CSV 输出；模型代码以 Jupyter Notebook 为主，辅以 .py 脚本
4. **沟通机制**：每 2-3 天在群内简短同步进度，遇到数据/内存问题及时沟通
5. **Commit 规范**：每次提交写明 `[A/B/C/D] 做了什么`，如 `[B] bureau 逾期聚合特征完成`
